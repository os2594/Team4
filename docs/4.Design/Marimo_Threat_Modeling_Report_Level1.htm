Description: The Marimo application provides an interactive, reactive notebook environment where users create, execute, and update Python code cells through a browser-based UI. All client actions such as cell execution, notebook state updates, file operations, and preview rendering are sent to the Marimo Backend Server through authenticated HTTPS API calls. The backend orchestrates execution in a controlled Python runtime ("execution sandbox"), manages notebook state, and returns results, rendered output, or error messages back to the user interface. The Backend Server manages state transitions, dependency tracking, and execution ordering of notebook cells. It also communicates with internal components responsible for saving notebooks, retrieving stored files, and packaging application artifacts. User code is executed inside an isolated environment to prevent unauthorized access to system resources. All data flowing between the frontend and backend crosses the Internet/Browser boundary, while internal file operations cross an internal storage boundary. The Level 1 diagram reflects these core architectural elements: External Web Client → Marimo Backend Server → Internal Storage Mechanisms, along with the associated trust boundaries that separate untrusted user input from controlled execution paths and system-level resources.

Assumptions: We assume that users interact with Marimo through a modern, standards-compliant web browser that supports secure HTTPS communication. All traffic between the browser and the Marimo backend is correctly encrypted, and users authenticate using supported mechanisms provided by the deployment environment. It is assumed that user code provided to the execution sandbox may be malicious and must never be trusted or executed with elevated privileges. We further assume that the host operating the Marimo backend is properly hardened, patched, and configured according to institutional or cloud security baselines. The execution sandbox is assumed to enforce proper isolation from host resources, preventing user code from escaping, accessing system files, or manipulating privileged processes. Finally, it is assumed that all internal file operations such as saving notebooks, retrieving application artifacts, or reading configuration occur within a controlled environment with appropriate access permissions enforced by the underlying OS or storage layer.

External Dependencies: The system depends on a functioning web browser and network stack on the user’s device to send and receive HTTPS traffic. It also depends on the hosting platform (cloud provider, VM, or container platform) to provide secure execution, networking, and filesystem services. If authentication or identity verification is used, Marimo relies on the external identity provider configured by the deployment environment. Additionally, the system relies on the Python runtime, its standard library, and installed packages for executing notebook code inside the sandbox. The correctness and security of external Python packages fall outside the direct control of Marimo and are considered external dependencies. Logging, monitoring, and optional storage backends (e.g., local filesystem or cloud storage) are also treated as external but required supporting services.

1. Cross Site Request Forgery  [State: Not Applicable]  [Priority: High] 

Category:Elevation Of Privilege
Description:Cross-site request forgery (CSRF or XSRF) is a type of attack in which an attacker forces a user's browser to make a forged request to a vulnerable site by exploiting an existing trust relationship between the browser and the vulnerable web site. In a simple scenario, a user is logged in to web site A using a cookie as a credential. The other browses to web site B. Web site B returns a page with a hidden form that posts to web site A. Since the browser will carry the user's cookie to web site A, web site B now can take any action on web site A, for example, adding an admin to an account. The attack can be used to exploit any requests that the browser automatically authenticates, e.g. by session cookie, integrated authentication, IP whitelisting. The attack can be carried out in many ways such as by luring the victim to a site under control of the attacker, getting the user to click a link in a phishing email, or hacking a reputable web site that the victim will visit. The issue can only be resolved on the server side by requiring that all authenticated state-changing requests include an additional piece of secret payload (canary or CSRF token) which is known only to the legitimate web site and the browser and which is protected in transit through SSL/TLS. See the Forgery Protection property on the flow stencil for a list of mitigations.
Justification:Not applicable: this is a server-to-server HTTPS API call using explicit auth tokens/keys, not a browser session with automatic cookies, so there is no authenticated user browser an attacker could trick into sending forged requests; CSRF is mitigated by not exposing this API directly to browsers and requiring signed API credentials on every call.

2. Elevation by Changing the Execution Flow in Backend Server  [State: Mitigation Implemented]  [Priority: High] 

Category:Elevation Of Privilege
Description:An attacker may pass data into Backend Server in order to change the flow of program execution within Backend Server to the attacker's choosing.
Justification: Enforce strong auth and per-endpoint authorization on all backend APIs, validate all control parameters, and run the external service under a least-privilege role so its credentials cannot invoke admin/privileged code paths.

3. Backend Server May be Subject to Elevation of Privilege Using Remote Code Execution  [State: Mitigation Implemented]  [Priority: High] 

Category:Elevation Of Privilege
Description:External Web Service may be able to remotely execute code for Backend Server.
Justification: The External Web Service sends authenticated inputs that the Backend Server parses, so a bug (e.g., injection or unsafe deserialization) could let crafted requests execute code with elevated privileges. This can be mitigated by never executing user data as code, using safe parsers/parameterized APIs, keeping frameworks patched, and running the backend under least-privilege accounts.

4. Elevation Using Impersonation  [State: Not Applicable]  [Priority: High] 

Category:Elevation Of Privilege
Description:Backend Server may be able to impersonate the context of External Web Service in order to gain additional privilege.
Justification:Not applicable: the Backend Server only receives authenticated API calls from the External Web Service and terminates that identity; it never forwards or reuses the service’s API credentials to other systems. All downstream calls use the Backend Server’s own fixed service account, so it cannot impersonate the External Web Service to gain additional privileges.

5. Data Flow API calls w/auth Is Potentially Interrupted  [State: Mitigation Implemented]  [Priority: High] 

Category:Denial Of Service
Description:An external agent interrupts data flowing across a trust boundary in either direction.
Justification:We mitigate this risk by using authenticated HTTPS/TLS for all cross-boundary communication, ensuring data cannot be intercepted or altered by an external agent.

6. Potential Process Crash or Stop for Backend Server  [State: Mitigation Implemented]  [Priority: High] 

Category:Denial Of Service
Description:Backend Server crashes, halts, stops or runs slowly; in all cases violating an availability metric.
Justification:We mitigate this risk by using redundancy, load balancing, health checks, and auto-scaling so that backend slowdowns or crashes do not impact overall system availability.

7. Data Flow Sniffing  [State: Mitigation Implemented]  [Priority: High] 

Category:Information Disclosure
Description:Data flowing across API calls w/auth may be sniffed by an attacker. Depending on what type of data an attacker can read, it may be used to attack other parts of the system or simply be a disclosure of information leading to compliance violations. Consider encrypting the data flow.
Justification:We mitigate this risk by enforcing HTTPS/TLS for all API calls, ensuring that data in transit is encrypted and cannot be sniffed or read by attackers.

8. Potential Data Repudiation by Backend Server  [State: Mitigation Implemented]  [Priority: High] 

Category:Repudiation
Description:Backend Server claims that it did not receive data from a source outside the trust boundary. Consider using logging or auditing to record the source, time, and summary of the received data.
Justification:The risk is mitigated because the backend server records every inbound request from outside the trust boundary using authenticated logging

9. Potential Lack of Input Validation for Backend Server  [State: Mitigation Implemented]  [Priority: High] 

Category:Tampering
Description:Data flowing across API calls w/auth may be tampered with by an attacker. This may lead to a denial of service attack against Backend Server or an elevation of privilege attack against Backend Server or an information disclosure by Backend Server. Failure to verify that input is as expected is a root cause of a very large number of exploitable issues. Consider all paths and the way they handle data. Verify that all input is verified for correctness using an approved list input validation approach.
Justification:Strict allowlist-based validation on all backend inputs prevents malformed or malicious data from altering backend behavior, blocking DoS, injection, and privilege-escalation attacks before they reach execution logic.

10. Spoofing the External Web Service External Entity  [State: Mitigation Implemented]  [Priority: High] 

Category:Spoofing
Description:External Web Service may be spoofed by an attacker and this may lead to unauthorized access to Backend Server. Consider using a standard authentication mechanism to identify the external entity.
Justification:Use mTLS or signed API credentials so the backend only accepts requests from authenticated external services. Certificate validation and strict allowlisting prevent attackers from impersonating the external web service and gaining unauthorized backend access.

11. Spoofing the Backend Server Process  [State: Mitigation Implemented]  [Priority: High] 

Category:Spoofing
Description:Backend Server may be spoofed by an attacker and this may lead to information disclosure by External Web Service. Consider using a standard authentication mechanism to identify the destination process.
Justification:Use TLS server certificates, strict certificate validation, and a reverse proxy as the sole entry point to authenticate the backend server process. Only the legitimate backend can present the correct certificate and validate signed session tokens. These controls prevent attackers from impersonating the backend and protect external services from disclosing data to a spoofed destination.

15. Spoofing of Destination Data Store Enterprise Internal Database  [State: Mitigation Implemented]  [Priority: High] 

Category:Spoofing
Description:Enterprise Internal Database may be spoofed by an attacker and this may lead to data being written to the attacker's target instead of Enterprise Internal Database. Consider using a standard authentication mechanism to identify the destination data store.
Justification:Implementation of session tokens/app ID/password applications to validate connection with database. The Backend Server executes untrusted notebook code, validates client requests using session tokens, and mediates all interactions across trust boundaries. A reverse proxy strengthens this boundary by enforcing TLS and filtering unauthorized traffic. Together, they form the security perimeter that isolates execution and protects host resources through authentication, authorization, and secure transport.

16. Potential SQL Injection Vulnerability for Enterprise Internal Database  [State: Mitigation Implemented]  [Priority: High] 

Category:Tampering
Description:SQL injection is an attack in which malicious code is inserted into strings that are later passed to an instance of SQL Server for parsing and execution. Any procedure that constructs SQL statements should be reviewed for injection vulnerabilities because SQL Server will execute all syntactically valid queries that it receives. Even parameterized data can be manipulated by a skilled and determined attacker.
Justification:Parameterized and santitized query enforcement and runtime detection prevent injection-based exfiltration.

17. Potential Excessive Resource Consumption for Backend Server or Enterprise Internal Database  [State: Mitigation Implemented]  [Priority: High] 

Category:Denial Of Service
Description:Does Backend Server or Enterprise Internal Database take explicit steps to control resource consumption? Resource consumption attacks can be hard to deal with, and there are times that it makes sense to let the OS do the job. Be careful that your resource requests don't deadlock, and that they do timeout.
Justification:A reverse proxy strengthens this boundary by enforcing TLS and filtering unauthorized traffic. Together, they form the security perimeter that isolates execution and protects host resources through authentication, authorization, and secure transport.

18. Spoofing the Backend Server Process  [State: Mitigation Implemented]  [Priority: High] 

Category:Spoofing
Description:Backend Server may be spoofed by an attacker and this may lead to unauthorized access to Enterprise Internal Database. Consider using a standard authentication mechanism to identify the source process.
Justification:The database requires to authenticate the backend server using strong credentials (TLS mutual certificates, Kerberos tickets).

19. The Enterprise Internal Database Data Store Could Be Corrupted  [State: Mitigation Implemented]  [Priority: High] 

Category:Tampering
Description:Data flowing across Parameterized queries may be tampered with by an attacker. This may lead to corruption of Enterprise Internal Database. Ensure the integrity of the data flow to the data store.
Justification:Use TLS 1.2/1.3 for all connections between backend and database. Apply cryptographic signatures (HMAC, digital signatures) to query payloads.

20. Data Store Denies Enterprise Internal Database Potentially Writing Data  [State: Mitigation Implemented]  [Priority: High] 

Category:Repudiation
Description:Enterprise Internal Database claims that it did not write data received from an entity on the other side of the trust boundary. Consider using logging or auditing to record the source, time, and summary of the received data.
Justification:Record every write operation with metadata: source identity, timestamp, query type, and affected tables/rows.

Justification:Encrypting the backend–database connection using TLS ensures parameterized query traffic cannot be intercepted or read. This protects sensitive query parameters and results from information disclosure and compliance violations.

22. Data Flow Parameterized queries Is Potentially Interrupted  [State: Mitigation Implemented]  [Priority: High] 

Category:Denial Of Service
Description:An external agent interrupts data flowing across a trust boundary in either direction.
Justification:Encrypt and authenticate all cross-boundary communications using TLS so that an external agent cannot read, modify, or disrupt data. Integrity checks, endpoint authentication, and connection validation ensure that only trusted parties can exchange data and that any tampering or interruption is detected and blocked.

23. Data Store Inaccessible  [State: Mitigation Implemented]  [Priority: High] 

Category:Denial Of Service
Description:An external agent prevents access to a data store on the other side of the trust boundary.
Justification:Access to the data store is protected by authentication, rate limiting, and network restrictions so that only authorized components can interact with it. Availability mechanisms such as retries, caching, and circuit breakers prevent an external agent from blocking or exhausting the data store, ensuring it remains accessible despite attempted denial-of-service attacks.

24. Spoofing of Source Data Store Enterprise Internal Database  [State: Mitigation Implemented]  [Priority: High] 

Category:Spoofing
Description:Enterprise Internal Database may be spoofed by an attacker and this may lead to incorrect data delivered to Backend Server. Consider using a standard authentication mechanism to identify the source data store.
Justification:Use strong, standardized authentication protocols (TLS with mutual certificate validation). Require encrypted connections (TLS 1.2/1.3) between backend and database. Configure the database to require unique credentials for the backend server.

25. Weak Access Control for a Resource  [State: Mitigation Implemented]  [Priority: High] 

Category:Information Disclosure
Description:Improper data protection of Enterprise Internal Database can allow an attacker to read information not intended for disclosure. Review authorization settings.
Justification:Assign permissions based on roles (analyst, admin, auditor). Ensure least privilege: users only access the data they need. Data classification (public, internal, confidential, restricted). Database encryption, files, backups, and logs.

26. Spoofing the Backend Server Process  [State: Mitigation Implemented]  [Priority: High] 

Category:Spoofing
Description:Backend Server may be spoofed by an attacker and this may lead to information disclosure by Enterprise Internal Database. Consider using a standard authentication mechanism to identify the destination process.
Justification:To mitigate this implement mutual TLS and server certificates that are signed by a trusted authority, and include some rotational mechanism to ensure that it is harder be spoofed. In addition, implement secure API keys and OAuth 2.0 for further hardening and authentication.

27. Potential Data Repudiation by Backend Server  [State: Mitigation Implemented]  [Priority: High] 

Category:Repudiation
Description:Backend Server claims that it did not receive data from a source outside the trust boundary. Consider using logging or auditing to record the source, time, and summary of the received data.
Justification:For mitigation enable verbose logging that can collect info such as source IP, timestamps, and hashes for authenticity. Also have it require the usage of digital signatures to confirm senders and source addresses.

28. Potential Process Crash or Stop for Backend Server  [State: Mitigation Implemented]  [Priority: High] 

Category:Denial Of Service
Description:Backend Server crashes, halts, stops or runs slowly; in all cases violating an availability metric.
Justification:For mitigation deploy high availability mechanisms and redundant instances to ensure data is always accessible. Auto scaling and load balancing could be stronger features to ensure all resources are being used efficiently with little to no crashes.

29. Data Flow Query results Is Potentially Interrupted  [State: Mitigation Implemented]  [Priority: High] 

Category:Denial Of Service
Description:An external agent interrupts data flowing across a trust boundary in either direction.
Justification:Encrypt all data in transit using TLS, and use digital signatures to confirm sender identity. VPN tunneling could be used for private networking.

30. Data Store Inaccessible  [State: Mitigation Implemented]  [Priority: High] 

Category:Denial Of Service
Description:An external agent prevents access to a data store on the other side of the trust boundary.
Justification:Restrict database resources and access to authorized users. Implement rate limiting to prevent overloading the database and server with too many requests at one time.

31. Backend Server May be Subject to Elevation of Privilege Using Remote Code Execution  [State: Mitigation Implemented]  [Priority: High] 

Category:Elevation Of Privilege
Description:Enterprise Internal Database may be able to remotely execute code for Backend Server.
Justification:The backend prevents elevation of privilege via remote code execution by enforcing strict separation of code and data, using parameterized queries, validating all database input, and running the backend with least-privilege roles. The database is restricted to read-only access and cannot trigger backend code execution paths, while execution isolation ensures the backend never interprets database content as executable instructions. Network segmentation prevents the DB from issuing backend commands, closing the RCE vector entirely.

32. Elevation by Changing the Execution Flow in Backend Server  [State: Mitigation Implemented]  [Priority: High] 

Category:Elevation Of Privilege
Description:An attacker may pass data into Backend Server in order to change the flow of program execution within Backend Server to the attacker's choosing.
Justification:The backend mitigates this threat by validating and sanitizing all inputs, ensuring that client data cannot alter control-flow decisions or be interpreted as executable code. Requests are constrained to strict API pathways, with no dynamic evaluation of user input, and runtime code execution is isolated in a restricted environment. Least-privilege backend roles and fixed routing logic prevent attackers from manipulating program flow or escalating privileges.
